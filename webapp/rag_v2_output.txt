A: Setting a Cubyz block's render mode to "cubyz:ore" triggers the engine-side ore properties for that block. This means that if an `.ore` block exists but has a rotation mode other than `"cubyz:ore"`, the engine logs `"Ore must have rotation mode \"cubyz:ore\"!"` and silently drops the ore properties, allowing the block to still load without its ore behavior.

Additionally, when setting the render mode to "cubyz:ore", it requires specific fields within the `.ore` block configuration. These include:
- `veins`: The average number of veins per chunk.
- `size`: The average vein size in blocks.
- `maxHeight`: The highest point the ore can generate.
- `minHeight`: The lowest point the ore can generate.
- `density`: The average density of each vein.

These fields are crucial for defining how the ore will behave and appear within the game world.

[Retrieved 13 chunks: github_issue_896_discussion.md, codebase_assets_cubyz_blocks_glass_cyan.zig.zon_chunk_0.md, codebase_assets_cubyz_blocks_glow_crystal_cyan.zig.zon_chunk_0.md, codebase_assets_cubyz_blocks_glow_crystal_indigo.zig.zon_chunk_0.md, codebase_assets_cubyz_blocks_glass_yellow.zig.zon_chunk_0.md, codebase_assets_cubyz_blocks_glow_crystal_dark_grey.zig.zon_chunk_0.md, codebase_assets_cubyz_blocks_glow_crystal_crimson.zig.zon_chunk_0.md, codebase_assets_cubyz_blocks_glow_crystal_lime.zig.zon_chunk_0.md, github_issue_2301_discussion.md, docs_CONTENT_SUGGESTIONS.md_chunk_0.md, docs_docs_development_addons_blocks.md_chunk_0.md, addon_creator_engine_validation_bug_skycolor.md, addon_creator_FIELD_REFERENCE.md_chunk_0.md]

A: The Cubyz Addon Creator converts a block's emitted/absorbed light color by using a helper function `getHexColorAsRGBVector` to transform the hexadecimal color code into a 3-component float vector (`.{r, g, b}`) with values ranging from 0-1. This conversion is then used in the exported `.zig.zon` file for the block's configuration.

However, there might be an issue where the engine reads these colors differently than expected due to a format mismatch between the website and the engine. The engine uses `zon.get(u32, "fogColor")` or `zon.get(u32, "skyColor")`, which expects packed integers (like `0xffbfe2ff`) rather than vectors. This could lead to silent fallbacks to default colors instead of crashing.

To ensure compatibility and avoid such issues, it's recommended that when hand-writing addon files, developers use the engine-expected packed `u32` color format even if the website's own export currently produces a vector instead.
   [expected: Hex color is converted to an integer via parseInt(hex, 16); only emitted if non-default]

======================================================================
[Retrieved 12 chunks: github_issue_2732_discussion.md, docs_automation_not_supported.md_chunk_0.md, docs_docs_faq.md_chunk_0.md, docs_docs_gameplay_controls.md_chunk_1.md, docs_docs_development_addons_index.md_chunk_0.md, docs_CONTENT_SUGGESTIONS.md_chunk_0.md, github_pr_3313_comment_3560517031.md, docs_docs_blocks_index.md_chunk_0.md, addon_creator_README.md_chunk_0.md, addon_creator_FIELD_REFERENCE.md_chunk_0.md, codebase_assets_cubyz_blocks_duckweed.zig.zon_chunk_0.md, codebase_assets_cubyz_blocks_osier.zig.zon_chunk_0.md]

A: In Cubyz, "auto" drop selection refers to a feature where items are automatically dropped from a block when it is broken. This means that instead of manually specifying which item should be dropped, the game uses predefined rules or conditions to determine what gets dropped.

On the other hand, manual drop selection allows for more control over what items can be dropped from a block. In this case, the user has to explicitly define which items and under what conditions they should be dropped. This gives more flexibility but requires more effort on the part of the user or developer creating the addon.

The difference between these two methods lies in how much control the user has over the drop process. "Auto" drops are simpler and require less configuration, while manual drops offer greater customization options but come with increased complexity.
   [expected: Auto exports as .{.items=.{.auto}} (drops the block itself); manual exports the specified item, auto-namespaced]

[85/144] Q: What's the structural difference between a Cubyz particle's type-level properties and its emitter-level properties, in terms of when they're validated?
======================================================================
[Retrieved 10 chunks: addon_creator_ENGINE_VALIDATION_REFERENCE.md_chunk_1.md, addon_creator_engine_validation_entities_particles.md, codebase_src_particles.zig_chunk_5.md, github_pr_3297_comment_3564114095.md, github_issue_3122_discussion.md, github_pr_1367_comment_2068718495.md, codebase_assets_cubyz_particles_flame.zig.zon_chunk_0.md, github_pr_1367_comment_2071284377.md, docs_docs_development_addons_blocks.md_chunk_0.md, docs_CONTENT_SUGGESTIONS.md_chunk_0.md]

A: The structural difference lies in the timing of validation. Type-level properties such as `.texture`, `.rotationVelocity`, `.density`, and `.dragCoefficient` are validated once at particle-type registration time (`register()`), which occurs when the addon loads. This means that if there's an error with these fields, it will be caught immediately during the registration process.

On the other hand, emitter-level properties like `.speed`, `.lifeTime`, `.shape`, `.mode`, and `.direction` are parsed separately whenever something actually spawns this particle type. This approach allows for a more flexible system where broken emitter-side fields won't cause an error at world-load time but will only surface when something in the world triggers that particle effect.

This distinction is important because it affects how errors are handled and reported within the engine. For instance, if there's an issue with a type-level property like `.texture`, the particle type fails to register entirely, resulting in a hard failure rather than a soft fallback. In contrast, issues with emitter-level properties only manifest when the particle effect is triggered, which can be less disruptive but still requires careful handling during runtime.
   [expected: Type properties (.texture/.rotationVelocity/.density/.dragCoefficient) are validated once, at registration time when the addon loads; emitter properties (.speed/.lifeTime/.shape/.mode/.direction) are parsed separately whenever something actually spawns the particle -- so a broken emitter field won't error at world-load time, only when triggered in-world]

[Retrieved 12 chunks: docs_docs_development_multiplayer.md_chunk_0.md, github_issue_2449_discussion.md, docs_docs_faq.md_chunk_0.md, docs_docs_development_multiplayer.md_chunk_1.md, github_issue_1681_discussion.md, github_issue_267_discussion.md, docs_docs_development_history.md_chunk_0.md, docs_docs_development_how_to_contribute.md_chunk_0.md, codebase_src_server_command__list.zig_chunk_0.md, codebase_assets_cubyz_sbb_mushroom_small_bolete.zig.zon_chunk_0.md, addon_creator_README.md_chunk_0.md, addon_creator_FIELD_REFERENCE.md_chunk_0.md]

A: Cubyz provides two main methods for hosting a multiplayer server:

1. **In-game Hosting**: This method allows players to host their own servers directly within the game, making it quick and easy to set up. However, it requires the game to keep running continuously; if the game is closed or restarted, all connected players will be disconnected.

2. **Headless Server (Dedicated)**: A dedicated server runs in the background without a graphical user interface. This setup is more suitable for long-term hosting and can be configured by setting specific parameters in the `launchConfig.zig` file, such as `.headlessServer = true` and `.autoEnterWorld = "<world name>"`.

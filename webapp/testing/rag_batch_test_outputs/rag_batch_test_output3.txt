[20/96] Q: Why might a biome's custom sky color not show up in-game even though I set it in the Addon Creator?
======================================================================
[Retrieved 12 chunks: addon_creator_engine_validation_bug_skycolor.md, codebase_assets_cubyz_biomes_volcano__defaults.zig.zon_chunk_0.md, codebase_assets_cubyz_biomes_rare_crimson_wasteland__defaults.zig.zon_chunk_0.md, codebase_assets_cubyz_biomes_cave_sky_islands__defaults.zig.zon_chunk_0.md, addon_creator_engine_validation_bugs.md, codebase_assets_cubyz_biomes_savannah__defaults.zig.zon_chunk_0.md, codebase_assets_cubyz_biomes_bog__defaults.zig.zon_chunk_0.md, codebase_assets_cubyz_biomes_desert_oasis__defaults.zig.zon_chunk_0.md, docs_docs_development_addons_biomes.md_chunk_0.md, docs_docs_development_addons_index.md_chunk_0.md, github_pr_2129_comment_2860728351.md, github_pr_1043_comment_1956745632.md]

A: A biome's custom sky color may not show up in-game because the Addon Creator website exports `.skyColor`/`.fogColor` as a 3-component float vector (`.{r, g, b}`, values 0-1) via `getHexColorAsRGBVector()`, while the engine reads them with `zon.get(u32, "fogColor")` / `zon.get(u32, "skyColor")`. The engine converts these packed integers to a vector internally via a separate `u32ToVec3()` helper. A vector literal read through a `u32` getter is very likely to fail to parse and silently fall back to the default color rather than crash. This is a format mismatch (vector vs. packed u32 integer), not a field-name mismatch.
   [expected: Format mismatch: the website exports skyColor/fogColor as a 3-component float vector, but the engine reads those fields as a packed u32 integer]

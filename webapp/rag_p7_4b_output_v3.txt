======================================================================
[Retrieved 12 chunks: docs_docs_development_history.md_chunk_0.md, docs_CONTRIBUTING.md_chunk_0.md, docs_README.md_chunk_0.md, github_issue_1954_discussion.md, docs_docs_development_modding.md_chunk_0.md, github_issue_118_discussion.md, github_issue_2609_discussion.md, github_issue_3329_discussion.md, addon_creator_FIELD_REFERENCE.md_chunk_0.md, addon_creator_ENGINE_VALIDATION_REFERENCE.md_chunk_0.md, codebase_assets_cubyz_sbb_tree_cirrus_vine.zig.zon_chunk_0.md, codebase_src_blueprint.zig_chunk_0.md]

A: Cubyz was rewritten from Java to Zig primarily due to issues with Java's garbage collector. When the upper memory limit was hit, Java would try to free memory, causing the game to freeze for a noticeable time. This lag spike was not the only complaint QuantumDeveloper had with Java. Before choosing Zig, he considered C++20 (rejected because its new "modules" feature wasn't supported by major compilers like GCC or Clang at the time) and Rust (rejected because found its borrow checker "kind of annoying").
   [expected: Java's garbage collector caused freeze-inducing lag spikes at the memory limit; Zig's cross-compilation support was also a major draw]

A: A Cubyz reviewer might suggest replacing manual string-splitting argument parsing with argparse because it offers several advantages over the previous method. The old approach, which involved manually splitting strings and comparing tokens against expected subcommand names using `std.mem.splitScalar(u8, args, ' ')`, was considered brittle due to its strict requirements for exact spacing between arguments (no tolerance for extra whitespace). Additionally, this method required a repeated string comparison for every possible subcommand token, leading to inefficiencies. Furthermore, it necessitated the creation of a custom per-command `Helper` struct that had to be manually initialized via `Helper.init(source, &split)` and explicitly cleaned up with a matching `deinit()` -- one more manual resource-management step a contributor could get wrong.

In contrast, argparse provides a generic parser utility that handles tokenization, per-field type parsing, and error-message construction itself. This allows individual commands to no longer duplicate the brittle logic of manual string-splitting argument parsing. By using a declarative `Args` union type to describe valid argument shapes (e.g., `/perm <action> <list> <playerIndex> <permissionPath>` vs. `/perm <playerIndex> <permissionPath>`), the new approach simplifies usage and improves code maintainability.

[114/144] Q: When is `catch unreachable` the correct way to handle error.OutOfMemory in Cubyz?
======================================================================
[Retrieved 12 chunks: docs_CONTRIBUTING.md_chunk_0.md, github_issue_1548_discussion.md, docs_CUBYZ_DEVELOPER_JUDGMENT.md_chunk_1.md, docs_CUBYZ_DEVELOPER_JUDGMENT.md_chunk_3.md, docs_docs_development_history.md_chunk_0.md, docs_CUBYZ_DEVELOPER_JUDGMENT.md_chunk_6.md, github_issue_630_discussion.md, docs_CUBYZ_DEVELOPER_JUDGMENT.md_chunk_9.md, addon_creator_engine_validation_blocks.md, addon_creator_ENGINE_VALIDATION_REFERENCE.md_chunk_0.md, codebase_src_server_command_worldedit_copy.zig_chunk_0.md, codebase_src_utils_heap.zig_chunk_2.md]

A: In Cubyz, `catch unreachable` should be used specifically for handling `error.OutOfMemory` errors when working with allocators that are structurally guaranteed not to fail, such as `NeverFailingAllocator`. This is because catching and handling an error that cannot occur implies a false possibility to future readers, which can lead to confusion or potential bugs.

On the other hand, using `unreachable` for cases where genuinely can happen under normal misconfiguration (like a missing structure reference) is considered a real bug. In such cases, it's recommended to replace `unreachable` with a proper error return and a logged message instead.

This distinction shows up repeatedly in the structure-building-block loader's evolution, where early versions would panic on a missing blueprint reference, and every round of review pushed toward "log an error and return null/error instead," because a bad addon config is not the same category of bug as a genuine engine invariant violation.
   [expected: When using the standard allocators (main.globalAllocator and main.stackAllocator), since error.OutOfMemory cannot happen with them]
